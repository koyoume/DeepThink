# 웹 기능 개발 가이드

> 웹 기반 도구/기능을 개발할 때 참조하는 실전 가이드. 기능 개발, repo 관리, 배포, 검증 방식을 다룬다.
> AssetTracker 웹 이식 프로젝트에서 실제로 겪은 성공·실패 사례가 근거. `DOC_GUIDE.md`(문서 체계 일반론)의 웹 프로젝트 특화판으로 봐도 된다.

---

## 0. 핵심 원칙

1. **계획 먼저, 결정 먼저, 작업은 그다음.** 애매한 설계 선택지는 추측하지 말고 짧게 확인받는다.
2. **문서 없는 진행은 없는 진행이다.** 세션(대화)이 끊기면 그 안의 기억은 사라진다 — md 파일과 git 커밋만 남는다.
3. **정본은 코드다.** 기존 앱을 포팅/참고할 땐 설계 문서보다 실제 소스 코드를 우선한다.
4. **막히면 재현부터.** 원인을 추측하기 전에 클린 환경에서 재현해 "내 코드 문제 vs 플랫폼/환경 문제"부터 가른다.
5. **환경별로 할 수 있는 일이 다르다.** 모바일 대화 세션과 데스크톱(Claude Code + Chrome 확장)은 역할이 다르다 — §7.
6. **UI 구조를 바꾸면 그 구조를 검사하던 테스트도 반드시 같이 바뀐다.** "기능은 고쳤는데 테스트는 그대로"는 배포 게이트에서 반드시 터진다.

## 1. 시작 전 체크리스트

- [ ] 참고할 기존 앱/코드가 있는가? → **그 코드를 직접 읽는다** (요약 문서 아님). 정본은 코드.
- [ ] 웹 프로젝트가 들어갈 저장소/폴더 위치 결정 (모노레포면 `/web` 같은 하위 폴더로 분리).
- [ ] 문서 2종 생성 — `PLAN.md`(진행상황·체크리스트·세션이력) / `REQUIREMENTS.md`(확정 스펙 누적). 이 저장소의 `DOC_GUIDE.md` 5파일 체계 중 PROGRESS+ARCHITECTURE+SETUP을 PLAN.md 하나로 압축한 웹 프로젝트용 경량판.
- [ ] 기술 스택 확정 → PLAN.md "확정된 결정" 표에 근거와 함께 기록.
- [ ] 배포 방식을 미리 정한다 — Cloudflare 쓸 거면 **§6을 먼저 읽는다** (시간 많이 쓴 함정 정리해둠).

## 2. 개발 워크플로우

### 2.1 계획 → 결정 → 작업

- 여러 그럴듯한 해석이 가능하고 결과가 크게 갈리는 지점은 코딩 전에 **1~3개짜리 짧은 객관식**으로 확인받는다.
  예) "차트 로그축에서 자산이 0이 되는 구간을 어떻게 표시할까? (A) 바닥에 붙여 선을 잇는다 (B) 선을 끊고 마커만 표시"
- 반대로 **기술적으로 명백한 수정**(빌드 에러, 버그)은 매번 물어보지 않고 바로 고치고 사후 보고한다.
- 결정 즉시 REQUIREMENTS.md에 반영. "나중에 정리"는 실제로는 안 지켜진다.
- **재작업 루프**: 사용자 피드백 → (애매하면) 계획 확인 → 구현 → 로컬 빌드/도메인검증 → commit & push → CI 결과 확인(가능하면) → 다음 피드백. 이 루프를 작은 단위로 여러 번 도는 게, 큰 덩어리로 한 번에 가는 것보다 안전하다.

### 2.2 세션이 끊길 수 있다는 전제로 일한다

- 작업 환경(특히 모바일 대화)은 **매번 초기화**될 수 있다. 커밋 안 한 변경은 없는 것과 같다.
- **의미 있는 단위마다 즉시 commit & push.** (실제로 1세션에서 문서 작성 중 세션이 끊겨 통째로 유실된 적 있음.)
- 채팅 중 만든 **미리보기 아티팩트도 세션 종료 시 사라진다.** 검토용으로 의미 있었다면 파일로 저장해 repo에 커밋(`web/doc/` 같은 폴더).

### 2.3 세션 인수인계 (여러 기기/사람이 동시에 작업할 때)

- 새 세션 시작 시 **`git fetch` 먼저** — 내가 모르는 커밋이 이미 있을 수 있다(다른 세션·다른 기기·데스크톱 Claude Code 등).
- 새 커밋을 발견하면 **diff를 실제로 읽고** 무슨 변경인지 파악한 후 이어간다.
- PLAN.md "세션 이력"에 시간순으로 기록 — 사실상 세션 간 유일한 "기억 저장소". **오판이 나중에 밝혀지면 조용히 넘기지 말고 정정 기록**을 남긴다(예: "전에 해결됐다고 했는데 사실 다른 에러에 가려져 있었을 뿐이었다").

## 3. 문서 체계 (PLAN + REQUIREMENTS 2종)

### `PLAN.md`
- 확정된 결정 표 (항목/결정/비고)
- 폴더 구조 (실제 반영본)
- 단계별 체크리스트 (P0, P1, P2 ...) — 완료 시 어떻게 검증했는지 한 줄
- 이 프로젝트 특화 워크플로 규칙 (예: "프로토타입은 항상 doc/에도 커밋")
- **세션 이력**: 시간순 로그, 정정 포함
- 열린 항목

### `REQUIREMENTS.md`
- 화면/기능별 확정 스펙을 **결정된 순서대로 누적**
- 코드 읽고 나중에 정정된 내용은 "정정" 표시와 함께 남긴다 (지우지 않음)
- 팔레트·단위·데이터 스키마처럼 계속 참조하는 것
- 구현 중 발견한 세부 스펙도 누적

**두 문서를 분리하는 이유**: "지금 뭘 해야 하는가"(PLAN)와 "정확히 뭘로 정했는가"(REQUIREMENTS)를 섞으면 금방 지저분해진다.

## 4. 기술 스택 선택 기준

- 기존 앱(안드로이드 등)을 웹으로 옮길 땐 **원본 디자인 시스템(색상·타이포)부터 조사**하고, 표준 라이브러리 팔레트와 매핑되는지 확인 — 실제로 원본 팔레트가 Tailwind 표준색과 거의 정확히 일치해 매핑표 하나로 색상 작업이 끝난 사례 있음. 새 디자인을 만들지 말고 원본 자산을 재사용.
- 이식 작업이면 "새롭고 독창적인 디자인" 가이드는 적용하지 않는다 — 목표는 원본과 일치.
- 프레임워크/빌드 도구는 **채팅 아티팩트 미리보기에서도 바로 확인 가능한 조합**을 우선 (예: Vite+React+Tailwind는 프로토타이핑과 실제 프로젝트 양쪽에서 거의 동일하게 동작).
- 의존성 버전은 **배포 플랫폼이 요구하는 최소 버전을 미리 확인** (안 그러면 배포 단계에서야 발견하게 됨).

## 5. 검증 전략

### 5.1 무엇을 어디서 검증하나

| 검증 대상 | 방법 | 실행 위치 |
|---|---|---|
| 순수 계산/도메인 로직 | 손계산 골든 케이스 + assert 스크립트 (`node`로 직접 실행) | 어디서나 |
| 빌드 성공 여부 | `npm run build` | 어디서나 |
| 배포 후 정적 파일 서빙 | HTTP 상태 코드 확인 | 어디서나 |
| **실제 브라우저 동작**(클릭·입력·화면이동) | E2E (Playwright) | **데스크톱/CI만** — 모바일 대화 샌드박스는 브라우저 바이너리 다운로드가 네트워크 정책으로 막혀 불가능 |
| 시각적 확인(레이아웃·터치감) | 사람이 직접, 또는 Claude in Chrome | 사용자 폰/데스크톱 |

### 5.2 도메인 로직 포팅 시 검증 기법

1. 원본 소스를 **라인 단위로 정독** (요약 문서 아님).
2. **1:1 포팅**하며 원본 언어의 미묘한 동작(정수 나눗셈 vs 부동소수, 타입별 반올림/절삭)을 그대로 재현 — 이런 디테일이 실제 버그 원인인 경우가 많음.
3. **손으로 계산한 골든 케이스**를 분기(경계조건·특수모드)별로 격리해서 여러 개 설계.
4. 검증 스크립트로 만들어 repo에 커밋(`__verify__/` 등) — 코드 바꿀 때마다 재실행해 회귀 방지.
5. 검증 스크립트가 **손계산 실수를 실제로 잡아낸 적 있다면** 제대로 작동 중이라는 신호.

### 5.3 E2E — 만드는 법과, UI가 바뀔 때 반드시 할 일

- 처음부터 완벽한 스위트를 짜지 않는다. **실사용 중 발견된 버그마다 재현 케이스를 추가**하며 키운다.
- 배포 파이프라인에 **배포 전 게이트**로 건다 — 실패하면 배포 자체가 안 되게.
- 날짜/시간 의존 계산은 **브라우저 시계를 고정**하고 실행해야 재현 가능.
- **UI 구조를 크게 바꾸면 그 구조를 검사하던 E2E도 반드시 같이 깨진다** — 예: "3년 주기 카드 목록"을 "1년 단위 차트"로 바꿨더니, 카드의 특정 CSS 클래스를 셀렉터로 쓰던 테스트와 카드 안 특정 텍스트("순증감:")를 찾던 테스트가 그대로 실패함. **UI를 바꿀 때 관련 E2E도 같은 커밋에서 같이 갱신**한다.
- 좌표 클릭처럼 **로컬에서 직접 실행해 확인할 수 없는 테스트 변경**(브라우저가 없는 환경에서 작성한 경우)은 실제 CI 실행 결과를 반드시 확인하고, 안 되면 즉시 재수정한다 — "짜놓고 안 돌려봄"은 배포 실패로 바로 드러남.
- 인터랙션 테스트는 **여러 후보 좌표/지점을 시도**하도록 짜면(하나라도 성공하면 통과) 좌표 추정이 살짝 틀려도 견고하다.

## 6. 배포 파이프라인

### 6.1 기본 원칙

- **CI(GitHub Actions 등)로 자동 배포.** 로컬 수동 배포는 모바일 환경에서 특히 번거롭다.
- 순서: **빌드 → (도메인/단위) 검증 → E2E 게이트 → 배포.** 앞 단계 실패 시 뒤로 안 감.
- 배포 후 **핵심 파일**(엔트리 HTML, manifest, 정적 자산 해시 파일, 아이콘)이 200으로 응답하는지 기계적으로 확인.

### 6.2 Cloudflare 관련 함정 (실제로 여러 시간 소모한 것들)

- **"Git 연동 → 자동 프레임워크 감지" 방식은 피한다.** 신규 프로젝트를 통합 "Workers" 배포로 자동 설정하면 `wrangler deploy`가 실행되는데, 이건 **`workers.dev` 서브도메인 등록을 강제**한다. 이 등록이 대시보드 UI 버그로 여러 경로(설정 탭, 신규 Worker 생성 등)에서 전부 막히는 사례가 실제로 있었음 — 심지어 "Hello World 템플릿 배포"처럼 완전히 다른 진입점도 똑같이 막혀서, 계정 자체의 문제임을 확정한 사례.
- **대신 `wrangler pages deploy`를 GitHub Actions에서 직접 실행**한다. `pages.dev` 도메인은 `workers.dev` 서브도메인 시스템과 완전히 무관해 위 문제를 원천 우회.
- `wrangler.jsonc`를 커밋할 때 **`assets.directory`를 반드시 포함**(빌드 산출물 경로). 이게 없으면 "missing the required directory property" 에러. Cloudflare 자동생성 스니펫을 그대로 복붙하면 이 필드가 빠져 있을 수 있음.
- 빌드 프레임워크의 최소 버전 요구사항을 미리 확인 (예: 특정 시점부터 Vite 6.0 미만은 자동 설정이 거부).
- **동일 에러가 반복되면, 클린 환경 재현으로 "내 캐시/설정 문제 vs 플랫폼 캐시 문제"부터 가른다** — 새 디렉토리에 clone → `npm ci` → 빌드 → 배포 dry-run까지 로컬에서 문제없음을 먼저 확인한 다음에야 "플랫폼 쪽 스테일 캐시" 같은 결론을 낸다.
- 계정 단위 1회성 설정(서브도메인 등록 등)이 막히면, **완전히 다른 배포 경로**(직접 업로드 등)를 함께 준비해둔다. 같은 UI를 계속 다르게 찔러보는 것보다 근본적으로 다른 경로가 있는지 먼저 확인.

### 6.3 시크릿 관리

- 배포 토큰은 **CI 시크릿 저장소**에 등록, 민감하지 않은 값(계정 ID 등)만 워크플로 파일에 상수로.
- 토큰이 대화/로그에 평문 노출됐다면 계속 쓰지 말고 **폐기 후 재발급**.

## 7. 작업 환경별 역할 분담

| | 모바일 Claude 대화 (샌드박스) | 데스크톱 Claude Code + Claude in Chrome |
|---|---|---|
| 코드 작성/리팩터링 | ✅ | ✅ |
| 빌드/도메인 로직 검증 | ✅ (`node` 직접 실행) | ✅ |
| 외부 API 호출 | ✅ 단, **화이트리스트 도메인만**(npm/pypi/github/apt 미러 등) | ✅ 제한 없음 |
| 임의 웹사이트 접속(curl/fetch) | ❌ 화이트리스트 밖은 막힘 | ✅ |
| 실제 브라우저 E2E 테스트 | ❌ 브라우저 바이너리 다운로드가 네트워크 정책으로 막힘 | ✅ |
| 배포 CLI 대화형 로그인(OAuth 등) | ❌ | ✅ |
| 실제 화면 클릭·입력·시각 확인 | ❌ | ✅ (Claude in Chrome) |
| 문서/커밋 관리 | ✅ | ✅ |

**실무 팁**: 코드 작성과 로직 검증은 모바일에서도 충분하다. **"배포 계정 로그인이 필요한 작업"과 "실제 브라우저로 확인해야 하는 작업"만 데스크톱으로 넘긴다.** 같은 git 저장소를 공유하니 한쪽이 막히면 넘기고 §2.3대로 인수인계.

## 8. 차트/시각화 구현 패턴

값의 범위가 크게 벌어지는 데이터(수십 년치 자산 성장 등)를 여러 계열 겹쳐 그릴 때 실제로 유효했던 패턴들.

- **로그 스케일 + 0(고갈) 처리**: 로그축에서 0은 수학적으로 표현 불가. "바닥값에 붙여 잇기"(부정확하지만 안 끊김) vs "그 지점에서 선을 끊고 마커만 표시"(정확하지만 안 이어짐) 중 하나를 반드시 사용자와 정하고 시작한다 — 애매하게 두면 나중에 다시 논의하게 된다.
- **Y축 하단을 고정값으로 두지 않는다.** 로그축 하단을 1 같은 고정값으로 두면 실제 데이터보다 훨씬 아래에서 시작해 화면 공간을 낭비한다. **현재 보이는 창의 실제 최솟값 기준으로 동적 계산**(약간의 여유 포함), 마커 등 별도 표시 요소가 잘리지 않게 하한을 보정.
- **범례 대신 라인 끝 직접 라벨링**: 계열이 많을 때 별도 범례 그리드보다, 각 선의 끝(보통 오른쪽)에 짧은 라벨을 직접 붙이는 쪽이 더 직관적일 수 있다. 라벨 겹침을 자동 보정할지(복잡도↑) 그냥 둘지(확대하면 자연히 벌어짐)는 미리 정한다.
- **색상만으로 부족하면 선 스타일도 곱한다.** 계열이 7개 이상이면 색상 팔레트만으론 구분이 어려움 — 대시 패턴(실선/파선/점선 등) 배열을 색상과 별도 인덱스로 곱해서 조합.
- **워터폴/스택 막대로 "증가·감소" 구성을 보여줄 때**: 감소 요소(지출 등)를 스택 맨 위에 두고 **빗금(해치) 패턴 fill**로 "이 부분은 빠진다"는 걸 시각적으로 표시. SVG `<pattern>`을 `<defs>`에 넣고 `fill="url(#id)"`로 참조 — 차트 라이브러리가 기본 지원 안 해도 이 방법으로 대부분 가능. 스택 차트는 **로그 스케일과 궁합이 안 맞음**(선형 스케일 사용).
- **드래그 팬 + 핀치 줌은 커스텀 구현이 필요할 수 있다.** 일반 차트 라이브러리는 터치 제스처를 기본 지원 안 하는 경우가 많음 — 네이티브 `touchstart/touchmove/touchend` 리스너(React 합성 이벤트 아님, `preventDefault`가 페이지 스크롤과 충돌 안 하게)로 "보이는 범위(윈도우)"를 직접 계산해 차트 라이브러리에 필터링된 데이터+도메인을 넘기는 방식으로 구현. **여러 차트가 같은 조작감을 공유해야 하면 이 로직을 훅으로 뽑아 재사용** — 그래야 한 곳만 고치면 다 같이 고쳐짐.
- **툴팁/범례 이름은 항상 내부 필드명이 아니라 사용자가 보는 라벨과 일치시킨다.** 차트 라이브러리는 기본적으로 `dataKey`(예: `stockGrowth`)를 그대로 툴팁에 표시하는 경우가 많음 — 각 계열에 `name`(표시용 라벨) 속성을 명시적으로 달아준다.
- **여러 차트가 한 앱에 있으면 그리드·여백·폰트 크기 스타일을 통일**해서 관리한다(공용 상수/스타일로 빼두면 나중에 일괄 변경이 쉬움).

## 9. 막혔을 때 접근 순서

1. **에러 메시지를 있는 그대로 읽는다.** 추측하지 말고 정확히 뭐라는지부터.
2. **내 쪽 문제인지 먼저 배제.** 클린 환경(새 clone, 새 디렉토리)에서 최소 재현. 로컬에서 문제없으면 "내 코드 문제는 아니다"까지 확정 가능.
3. 그래도 안 되면 **같은 증상을 검색**, 단 결과의 **날짜**를 확인(플랫폼 이슈는 금방 고쳐지기도 함).
4. **한 우회로가 안 되면 근본적으로 다른 경로**가 있는지 찾는다. 같은 UI를 계속 다르게 찔러보는 건 시간 낭비일 수 있음.
5. **오판이 밝혀지면 조용히 넘어가지 말고 정정 기록**을 남긴다.
6. **사람이 직접 계정 설정을 해야 하는 부분(로그인, 결제, 도메인 등록 등)은 대신할 수 없다는 걸 빨리 인정**하고, 정확히 어디를 눌러야 하는지 안내하는 데 집중한다. 모바일 UI는 데스크톱 기준 관리자 화면과 다르게 렌더링될 수 있음도 염두에.

## 10. 새 프로젝트 시작 시 복사해서 쓰는 골격

### `PLAN.md` 골격
```markdown
# <프로젝트명> — 계획·진행 현황
> 상태: <현재 단계> · 최종 갱신 <날짜>

## 1. 확정된 결정
| 항목 | 결정 | 비고 |
|---|---|---|

## 2. 폴더 구조

## 3. 단계별 작업 (체크리스트)
- [ ] P0 · ...

## 4. 개발 워크플로우

## 5. 열린 항목 / 추후 결정

## 6. 세션 이력 (진행 로그)
- **<날짜> (세션 N)**: ...
```

### `REQUIREMENTS.md` 골격
```markdown
# <프로젝트명> — 요구사항 (누적)
> 정본은 코드. 이 문서는 그 위에 얹는 결정·적응 사항을 담는다.

## 1. 단위/용어
## 2. 데이터 모델
## 3. 화면/기능별 스펙
## 4. 비기능 요구 (성능, 오프라인 등)
## 5. 범위 제외 (Non-Goals)
## 6. 구현 메모 (실제 코딩하며 발견/정정한 것)
```

### GitHub Actions 배포 워크플로 골격 (Cloudflare Pages 예시)
```yaml
name: Deploy Web
on:
  push:
    branches: [main]
    paths: ['web/**']
  workflow_dispatch: {}
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '22' }
      - run: npm ci
        working-directory: web
      - run: npm run build
        working-directory: web
      - run: node src/domain/__verify__/cases.mjs   # 있다면
        working-directory: web
      - run: npm run test:e2e                        # 있다면, 배포 전 게이트
        working-directory: web
      - run: npx wrangler pages deploy ./dist --project-name <project> --branch main
        working-directory: web
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}
```

## 11. 안티패턴 (하지 말 것)

- ❌ 세션 끝날 때까지 커밋을 미루기
- ❌ 애매한 UX 결정을 혼자 추측해서 구현부터 하기
- ❌ 요약 문서만 보고 원본 코드를 안 읽고 포팅하기
- ❌ 배포 에러가 나면 바로 "플랫폼 버그"로 단정하기 (클린 재현으로 내 쪽부터 배제한 다음에)
- ❌ 채팅 아티팩트로 만든 프로토타입을 파일로 저장 안 하고 다음 세션에서 다시 만들기
- ❌ 같은 막힌 UI를 계속 다른 방식으로 찔러보기만 하고 대안 경로를 안 찾기
- ❌ 모바일 샌드박스에서 안 되는 걸(실브라우저 E2E, 임의 도메인 접속 등) 될 때까지 계속 시도하기 — 안 되는 이유를 확인했으면 인정하고 적절한 환경(데스크톱)으로 넘긴다
- ❌ **UI 구조를 바꾸면서 그 구조를 검사하던 E2E 테스트를 안 고치기** — 배포 게이트에서 반드시 걸림
- ❌ 로컬에서 실행 확인 못 한 테스트 변경을 "짜놨으니 됐다"고 커밋만 하고 CI 결과를 안 보기
